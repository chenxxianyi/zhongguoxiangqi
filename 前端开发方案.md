# 棋境（Xiangqi Lab）前端开发方案

> 本方案依据根目录 `xiangqi-ai-codex-prompt.md` 拆分，面向响应式 Web/PWA 的前端研发。前端负责交互、展示、状态同步和用户体验，不作为中国象棋规则、对局状态或 AI 决策的权威来源。

## 1. 项目目标

前端需要交付一套适配桌面端与移动端的中国象棋产品界面，覆盖以下核心闭环：

1. 创建人机对局并完成一盘棋；
2. 实时展示玩家落子、AI 思考、计时和对局结果；
3. 上传棋谱并查看解析、去重和失败报告；
4. 创建、查看、启用和回滚学习库版本；
5. 查看历史对局、评估曲线、关键转折和学习库命中情况；
6. 查看引擎健康、资源状态、隐私与许可证信息。

MVP 首选响应式 Web/PWA，不因 Tauri 或其他桌面壳延期，但工程结构和浏览器 API 使用方式应兼容后续桌面封装。

## 2. 前端职责边界

### 2.1 前端负责

- 页面布局、交互流程、路由与权限入口；
- 棋盘绘制、棋子拖拽/点击、动画和提示；
- 调用 REST API，维护请求加载、成功、失败和重试状态；
- 订阅 WebSocket 事件并根据版本号同步对局状态；
- 展示服务端返回的合法着、FEN、记谱、评分、候选着和分析数据；
- 文件上传、进度、批次状态和结构化错误展示；
- PWA、响应式、无障碍、国际化预留和本地偏好设置；
- 前端单元测试、组件测试和关键 E2E 测试。

### 2.2 前端不负责

- 不独立裁定走法是否合法、是否将军、将死或重复局面；
- 不自行生成权威中文记谱、局面哈希或对局结果；
- 不直接连接或控制 Pikafish 等外部引擎进程；
- 不在浏览器中执行用户上传的棋谱或可执行文件；
- 不伪造“棋谱训练完成”、精确 Elo、评估数据或引擎状态；
- 不以本地状态覆盖服务端更高版本的对局状态。

前端可以基于服务端下发的合法着做即时高亮，但玩家落子仍必须提交服务端再次验证。

## 3. 技术栈与关键决策

### 3.1 推荐技术栈

- Vue 3（Composition API + `<script setup>`）；
- TypeScript，开启严格模式；
- Vite；
- Vue Router；
- Pinia；
- 原生 `fetch` 的统一封装，或选择轻量 HTTP 客户端；
- WebSocket 原生 API + 自建重连层；
- Vitest + Vue Test Utils；
- Playwright；
- ESLint + Prettier；
- PWA 插件用于 Service Worker、Manifest 和更新提示。

### 3.2 棋盘实现选择

优先使用 **SVG** 绘制棋盘和棋子，理由如下：

- 9×10 交叉点坐标容易精确映射；
- 缩放清晰，适合高 DPI 和移动端；
- 可为棋子、落点、最后一步、将军提示提供语义和可访问属性；
- 拖拽、点击、键盘焦点和动画都较易控制；
- 后续导出棋盘快照或在桌面壳中复用更方便。

不建议 MVP 首选 Canvas：Canvas 在无障碍、自动化测试和单元素交互方面成本更高。若后期性能基准表明 SVG 成为瓶颈，再以可替换渲染层优化。

### 3.3 状态权威策略

- 服务端对局快照是唯一权威状态；
- 每个快照携带 `matchVersion`；
- 前端提交落子时携带当前版本和幂等键；
- 只接受版本不低于本地版本的新状态；
- WebSocket 断线重连后先拉取最新快照，再补齐事件；
- 乐观交互仅表现为“待确认落子”，服务端拒绝后立即回滚并展示原因。

## 5. 视觉设计与设计系统

### 5.1 风格定位

采用“现代东方棋院”风格：

- 主背景：温润纸色；
- 主文字：深墨色；
- 强调色：克制朱砂红；
- 辅助色：少量木色、灰绿色或靛青；
- 避免大面积渐变、霓虹、重阴影、过度玻璃拟态；
- 不堆砌龙纹、卷轴、金边等俗套古风元素；
- 棋盘始终是对弈页的视觉中心。

### 5.2 设计令牌

在 `tokens.css` 统一维护：

- 颜色：背景、文字、边框、危险、成功、选中、合法落点、最后一步、将军；
- 字号与行高；
- 间距阶梯；
- 圆角和阴影等级；
- 动效时长与缓动；
- 桌面、平板、手机断点；
- 棋盘尺寸计算变量。

状态提示不能只依赖颜色，需同时使用形状、描边、图标或文本，确保色盲用户可区分。

## 6. 核心页面方案

### 6.1 首页

功能：

- 继续未完成对局；
- 快速开始；
- 最近战绩；
- 当前启用学习库摘要；
- 棋谱导入/学习任务快捷入口；
- 引擎异常时显示非阻断式状态提示。

桌面端使用信息分区而非全部卡片化；移动端优先“继续对局”和“快速开始”。

### 6.2 新对局设置页

配置项：

- 执红、执黑、随机；
- 难度 1～10，并映射为入门、休闲、进阶、高手、大师；
- AI 模式：标准引擎、棋谱库优先、棋风模仿；
- 学习库版本；
- 棋风画像；
- 思考时间/对局限时；
- 是否允许悔棋；
- 先手和随机种子相关说明（仅调试模式可固定种子）。

若外部引擎不可用，前端展示“已降级为内置引擎”，但仍允许支持的难度开始对局。

### 6.3 对弈页

核心区域：

- SVG 棋盘；
- 双方名称、执色、计时和当前轮次；
- AI 思考状态；
- 着法列表；
- 悔棋、认输、求和、翻转棋盘、坐标、音效开关；
- 网络断开、恢复、服务器拒绝、引擎降级提示。

交互要求：

- 同时支持点击走子、拖拽走子、触控和键盘；
- 选中棋子后显示服务端提供的合法落点；
- 最后一步、选中、可走、吃子、将军使用不同视觉编码；
- 落子提交后锁定重复操作，直到收到接受/拒绝或新版本状态；
- AI 思考时允许浏览历史着，但不能修改权威当前局面；
- 悔棋后清除旧 AI 思考展示，等待新版本状态；
- 移动端将着法、对局信息和次要操作收纳到抽屉。

### 6.4 棋谱库

功能：

- 拖拽/选择上传一个或多个支持文件；
- 上传前显示类型、大小和数量校验；
- 展示上传、解析、规范化、去重等阶段进度；
- 批次报告显示总数、成功、重复、失败、警告和未知结果；
- 错误精确到局号、回合、原文片段和原因；
- 棋谱详情支持逐着播放、元信息、标签和规范记谱；
- 有歧义的中文记谱只展示服务端候选解释，不静默确认；
- 删除操作明确说明对已发布学习版本的影响。

### 6.5 学习中心

功能：

- 选择棋谱集合；
- 配置最小样本数、近期权重、温度等允许开放的参数；
- 创建异步学习任务；
- 展示任务阶段、进度、失败原因和重试状态；
- 展示有效棋谱/着数、重复率、非法着率、局面覆盖、开局分布、胜和分布、低样本量和棋风置信度；
- 查看棋风画像；
- 启用新版本和回滚旧版本；
- 明确版本不可变，启用状态可切换。

### 6.6 复盘页

功能：

- 局面逐着播放；
- 评分曲线；
- 实战着、最佳着和 MultiPV 候选；
- 关键转折、失误等级和评分损失；
- 开局库命中、AI 使用学习着比例；
- 玩家重复失误和薄弱开局提示；
- 分析未完成时展示真实进度，不使用伪数据占位。

图表需支持键盘访问和文本摘要，移动端以棋盘为主，曲线和候选变化放入抽屉或分页。

### 6.7 历史对局

- 按结果、难度、AI 模式、日期和复盘状态筛选；
- 显示胜/负/和、执色、用时、学习库版本和命中率；
- 支持继续未完成对局、打开复盘和导出自有数据。

### 6.8 设置与诊断

- 棋盘方向、坐标、音效、动画和主题偏好；
- 外部/内置引擎健康状态；
- 引擎名称、版本和当前降级状态；
- 数据导出与隐私说明；
- 关于页面和第三方许可证；
- 生产环境不展示敏感路径、原始引擎日志、内部指标或 `pprof` 信息。

## 7. 棋盘组件设计

### 7.1 组件拆分

```text
XiangqiBoard.vue
├─ BoardGrid.vue
├─ BoardCoordinates.vue
├─ BoardPiece.vue
├─ MoveHints.vue
├─ LastMoveMarker.vue
├─ CheckIndicator.vue
└─ BoardStatusOverlay.vue
```

组件命名和领域术语必须使用中国象棋概念，避免引入国际象棋的升变等无关模型。

### 7.2 输入输出

主要输入：

- `position`：服务端返回的棋子与规范坐标；
- `orientation`：red/black；
- `selectedSquare`；
- `legalMoves`；
- `lastMove`；
- `checkSide`；
- `disabled`；
- `pendingMove`。

主要事件：

- `select-square`；
- `attempt-move`；
- `flip`；
- `request-legal-moves`。

`attempt-move` 仅表达用户意图，不能在组件内部直接永久修改局面。

### 7.3 坐标规范

- API 层始终使用服务端定义的 `fromFile/fromRank/toFile/toRank` 或 ICCS 字符串；
- 红黑翻转只发生在 `domain coordinate -> display coordinate` 映射层；
- SVG 尺寸、点击区域和棋子中心均由同一套计算函数生成；
- 坐标转换函数必须有红/黑视角往返单测。

## 8. 前端状态管理

### 8.1 Pinia Store 划分

- `matchStore`：当前对局快照、版本、合法着、计时、连接状态、待确认操作；
- `recordsStore`：上传队列、批次、棋谱列表和筛选；
- `learningStore`：学习任务、版本、质量报告和启用状态；
- `analysisStore`：分析进度、评分序列、候选着和关键转折；
- `preferencesStore`：棋盘方向、坐标、音效、动效和布局偏好；
- `authStore`：用户与会话信息（若首版引入账号体系）。

服务端业务数据和纯 UI 状态应分离，避免一个全局 Store 承担所有职责。

### 8.2 对局状态机

前端可使用以下展示状态：

```text
idle -> creating -> player_turn -> submitting_move -> ai_thinking
ai_thinking -> player_turn | finished | reconnecting | recoverable_error
player_turn -> undoing | resigning | offering_draw | finished
```

该状态机用于控制界面，不替代服务端对局状态机。任何冲突以服务端快照为准。

## 9. API 与 WebSocket 集成

### 9.1 REST 客户端

统一处理：

- API Base URL；
- JSON 编解码；
- 超时与取消；
- 身份凭证；
- `Idempotency-Key`；
- 结构化错误映射；
- 关联请求 ID；
- 401/403/409/422/429/5xx 的差异化反馈。

写接口不得在网络超时后盲目重复提交，应使用相同幂等键查询或重试。

### 9.2 WebSocket 事件

事件至少包含：

```ts
interface MatchEvent<T = unknown> {
  eventId: string
  matchId: string
  matchVersion: number
  type: string
  timestamp: string
  payload: T
}
```

客户端策略：

1. 建立连接时携带最后已确认的 `eventId` 或版本；
2. 心跳超时后进入重连状态；
3. 指数退避并加入随机抖动；
4. 重连后请求最新对局快照；
5. 丢弃旧版本、重复事件和不属于当前对局的事件；
6. 页面隐藏后降低非必要刷新频率，但不得错误暂停服务端计时；
7. 多标签页同时操作时，过期标签页收到版本冲突后转为只读并刷新。

## 10. 错误、空状态与降级体验

必须设计以下真实状态：

- 首次无历史对局；
- 无棋谱、无学习版本、无复盘；
- 上传文件过大、格式不支持、编码失败、部分棋局非法；
- Redis/任务队列暂不可用；
- 外部引擎不可用并降级到内置引擎；
- AI 搜索超时、取消或结果过期；
- WebSocket 断线但 REST 仍可用；
- 对局版本冲突；
- 学习版本质量检查未通过；
- 分析任务失败并允许重试。

错误提示应包含“发生了什么、当前数据是否安全、用户下一步能做什么”，不直接向普通用户暴露堆栈或内部路径。

## 11. 响应式与无障碍

### 11.1 响应式

- 桌面端：棋盘居中，右侧显示着法和对局信息；
- 平板端：棋盘优先，信息区域可折叠；
- 手机端：棋盘接近可用宽度，计时和核心操作常驻，着法/分析进入底部抽屉；
- 避免固定像素导致 9×10 棋盘变形；
- 横屏与竖屏均可完成对局；
- 触控目标不小于合理可点击尺寸，拖拽失败时仍可点击走子。

### 11.2 无障碍

- 棋盘提供可聚焦的交叉点/棋子；
- 键盘可选择棋子、移动焦点并提交走法；
- 使用 `aria-live` 播报走子、将军、对局结果和连接变化；
- 图表提供文本摘要；
- 所有图标按钮有可读标签；
- 支持减少动态效果偏好；
- 保证文字和状态标识的对比度。

## 12. 性能方案

- 路由级代码分割；
- 棋谱列表、历史列表使用虚拟滚动或服务端分页；
- 大型分析数据按需加载；
- 评分曲线增量渲染，避免每个事件重建整图；
- 对局页避免深层响应式大对象，按版本替换不可变快照；
- 音频资源预加载但不阻塞首屏；
- Service Worker 不缓存包含隐私数据的 API 响应；
- AI 思考完全在服务端执行，不占用浏览器主线程；
- 通过浏览器性能测量验证优化，避免无基准微调。

建议验收指标：

- 常用移动设备上棋盘点击/拖拽无明显卡顿；
- 路由切换和首屏骨架及时可见；
- 对局事件到达后尽快完成局面更新；
- 10,000 盘棋谱的前端列表依靠分页/虚拟化保持可用，而非一次加载全部记录。

## 13. 安全与隐私

- 对用户可见文本默认转义，不渲染不可信 HTML；
- 文件上传前端校验只用于体验，服务端仍执行完整安全校验；
- 不保存外部引擎二进制路径或服务端敏感配置；
- Token 优先使用安全 Cookie 策略，避免长期敏感凭证暴露于本地存储；
- 导出、删除和版本切换需二次确认；
- 日志与埋点不记录完整棋谱内容、凭证或敏感用户数据；
- PWA 缓存策略区分静态资源和用户数据；
- 关于/许可证页准确展示外部引擎来源与许可证，不把第三方引擎称为自研。

## 14. 测试方案

### 14.1 单元测试

- 红黑视角坐标映射往返；
- ICCS/结构化坐标展示转换；
- 对局版本事件去重和乱序处理；
- 难度分组、状态文案和数据格式化；
- 幂等键生成与重试逻辑；
- 上传队列状态流转；
- Pinia Store 状态机。

### 14.2 组件测试

- 棋盘点击和拖拽发出正确走子意图；
- 选中、合法点、最近一步和将军提示；
- 服务端拒绝非法着后的回滚；
- AI 思考期间禁用重复落子；
- 移动端抽屉和核心操作；
- 上传报告的成功、重复、失败与警告状态；
- 评分曲线和关键转折的空/加载/成功/失败状态。

### 14.3 E2E

至少覆盖：

1. 创建对局 → 玩家走子 → AI 回应 → 悔棋；
2. 断开 WebSocket → 自动重连 → 恢复最新对局版本；
3. 上传棋谱 → 查看导入报告；
4. 创建学习版本 → 查看质量报告 → 启用；
5. 使用学习库开始对局 → 查看学习库命中标记；
6. 完成对局 → 创建复盘 → 查看关键转折；
7. 外部引擎不可用 → 界面显示降级且可恢复；
8. 手机视口完成一盘对局的核心操作。

测试中使用可控测试后端或真实集成环境，不以静态假数据代替关键业务验证。

## 15. 分阶段实施计划

### 阶段 F0：脚手架与设计基础

交付：

### 阶段 F1：规则验证棋盘

- SVG 棋盘、棋子、坐标和翻转；

- 服务端局面与合法着展示；
- 本地双人对局验证页面；
- 坐标、交互和组件测试。

### 阶段 F2：基础人机对战

交付：

- 新对局设置；
- 对局状态、计时、着法列表；

## 16. 前后端接口协作清单

后端应优先提供以下契约，前端可据此并行开发：

1. 坐标、棋子、FEN、Move、MatchSnapshot 的统一 JSON Schema；
2. 错误响应结构和稳定错误码；
3. 对局状态枚举和结束原因枚举；
4. WebSocket 事件类型、版本规则和重连方式；
5. 难度配置中哪些字段对普通用户可见；
6. 上传限制、支持格式和导入错误结构；
7. 学习任务、版本、质量指标和棋风画像结构；
8. 复盘评分单位、红方视角约定、Mate 分数表达；
9. 时间字段统一使用 UTC ISO 8601，前端按本地时区展示；
10. 分页、筛选、排序和幂等策略。

建议在 `docs/api/` 或 OpenAPI 文档中维护契约，并由 CI 生成 TypeScript 类型或校验契约兼容性。

## 17. 前端完成标准

前端方案完成需同时满足：

- 桌面和移动浏览器均可开始并完成一盘人机对局；
- 棋盘 9×10 比例正确，红黑视角和坐标映射经过测试；
- 点击、拖拽、触控和键盘操作可用；
- 非法着由服务端拒绝，前端能正确回滚和提示；
- WebSocket 断线可恢复，旧版本事件不会污染当前状态；
- 可以上传棋谱并查看逐项导入报告；
- 可以创建、启用和回滚学习版本；
- 可以查看历史、基础复盘、关键转折和学习库命中率；
- 外部引擎异常时显示真实降级状态；
- 响应式、无障碍、单元测试、E2E、静态检查和构建全部通过；
- 无假数据冒充已实现能力，无硬编码密钥，无敏感信息泄漏。

## 18. 主要风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 前后端坐标约定不一致 | 棋子翻转、落点错误 | 共享契约与固定测试夹具，集中实现显示转换 |
| 前端复制规则后与后端分叉 | 出现“界面可走但服务端拒绝” | 前端只消费服务端合法着，服务端始终最终校验 |
| WebSocket 乱序/重复 | 棋盘回退或重复落子 | `eventId + matchVersion` 去重，重连拉快照 |
| 移动端信息过载 | 棋盘过小、操作困难 | 棋盘和计时优先，次要信息进入抽屉 |
| 大棋谱和分析数据卡顿 | 列表和图表不可用 | 分页、虚拟化、按需加载、增量渲染 |
| 引擎/任务失败被误认为页面故障 | 用户无法判断数据是否安全 | 明确状态、降级说明、重试和诊断入口 |
| 许可证表述错误 | 分发和品牌风险 | 关于页从后端/构建清单读取版本与许可证信息 |
| 过度追求古风装饰 | 降低可用性和现代感 | 坚持设计令牌、克制配色和棋盘主体地位 |

---

本方案的实施顺序应始终服从项目总体优先级：**规则正确性 → 对局状态同步 → 可控 AI → 棋谱规范化与学习 → 复盘与产品化**。
