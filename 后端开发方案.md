# 棋境（Xiangqi Lab）后端开发方案

> 本方案依据根目录 `xiangqi-ai-codex-prompt.md` 拆分，面向 Golang 后端、规则核心、对弈引擎、棋谱管线、学习库和复盘系统研发。服务端是规则、对局状态、局面哈希、引擎结果校验和数据完整性的唯一权威来源。

## 1. 项目目标

后端需要提供一套独立、可测试、可降级的中国象棋服务，形成以下完整能力：

1. 正确表达棋盘、走法、FEN、坐标、记谱、将军和终局；
2. 支持人机对局状态机、计时、悔棋、认输、求和和断线恢复；
3. 提供内置基础搜索引擎，并可选接入 Pikafish 等外部强引擎；
4. 提供可解释、可配置、可复现验证的 1～10 级难度系统；
5. 安全导入棋谱，逐着回放验证、规范化、去重并生成报告；
6. 从棋谱构建版本化局面着法库和棋风画像；
7. 对局时按规则、学习库、引擎、难度和棋风的固定顺序决策；
8. 异步生成复盘、关键转折、候选着和用户薄弱项统计；
9. 在 Redis 或外部引擎不可用时可靠降级，不损坏对局；
10. 通过 Docker Compose、CI、测试、指标和文档形成可持续研发工程。

MVP 不实现或宣称 NNUE 再训练、强化学习、自我对弈集群或大模型“学会下棋”。

## 2. 当前仓库评估

当前根目录仅包含需求文档，尚无源码、README、依赖、数据库迁移或未提交实现。因此建议按前后端分离 monorepo 从零初始化，并优先完成规则核心的可运行纵向切片。

本方案不直接决定第三方引擎二进制的分发方式。正式集成和打包 Pikafish 前必须确认 GPLv3 合规策略。

## 3. 后端职责边界

### 3.1 后端负责

- 所有规则计算、合法着生成、终局和重复局面裁决；
- 规范坐标、FEN、中文展示记谱和 Zobrist 哈希；
- 权威对局状态、版本号、计时和并发控制；
- AI 搜索、学习库命中、候选着过滤、棋风重排和难度采样；
- 外部引擎进程生命周期、超时、取消、池化和降级；
- 棋谱上传安全、解析、逐着验证、去重和持久化；
- 学习版本构建、质量检查、启用和回滚；
- 异步任务、失败重试、审计、指标和结构化日志；
- API、WebSocket、鉴权、限流、幂等和数据隔离；
- MySQL、Redis 和对象存储抽象。

### 3.2 后端不负责

- 不信任前端传入的合法着、局面哈希、当前轮次或对局结果；
- 不执行用户上传的棋谱文件；
- 不允许用户上传或指定外部引擎可执行文件；
- 不把外部引擎返回的走法直接写入对局，必须再次通过规则层；
- 不伪造精确 Elo、学习效果、复盘评分或任务完成状态；
- 不在许可证策略未确认时默认提交或分发第三方引擎二进制。

## 4. 技术栈与工程决策

### 4.1 推荐技术栈

- Go：使用实施时的稳定版本并通过 `go.mod`/toolchain 锁定；
- HTTP：优先 `net/http`，可配合轻量路由器；
- 数据库：MySQL 8.x，显式 SQL，迁移工具管理 schema；
- Redis：任务队列、短期状态、限流和分布式协调，本地允许内存降级；
- 实时通信：WebSocket；
- 对象存储：定义接口，本地文件系统实现用于开发，生产接 S3/MinIO；
- 观测：结构化日志、Prometheus 指标、OpenTelemetry 预留、受控 `pprof`；
- 测试：Go 原生 testing、属性测试/模糊测试、benchmark、集成测试；
- 工程：Docker Compose、Makefile、GitHub Actions 或等价 CI。

### 4.2 架构风格

采用分层模块化单体作为 MVP：

```text
transport -> application/service -> domain
                         \-> repository ports -> MySQL/Redis/storage
                         \-> engine ports -> builtin/external
                         \-> async job ports -> worker
```

理由：

- 规则、引擎、导入和学习已经足够复杂，首版不宜再引入微服务运维成本；
- API 与 worker 可以是不同进程，但共享稳定的领域和应用模块；
- 后续若任务规模增长，可按边界拆分服务；
- 通过接口隔离数据库、Redis、对象存储和外部引擎，便于测试和降级。

### 4.3 一致性策略

- MySQL 保存长期权威数据；
- Redis 保存可重建的短期状态和队列信息，不作为唯一事实来源；
- 对局每次变更使用单调递增 `version`；
- 写操作使用事务、唯一约束和乐观锁；
- AI 搜索结果写回前校验 `match_id + expected_version + side_to_move`；
- 旧结果、重复请求和已取消任务不得修改当前局面；
- 重要异步事件建议使用事务型 Outbox 或等价机制，避免数据库提交成功但任务消息丢失。

## 6. 中国象棋规则核心

### 6.1 领域模型

建议定义：

- `Color`：Red、Black；
- `PieceType`：General、Advisor、Elephant、Horse、Rook、Cannon、Pawn；
- `Piece`：颜色 + 类型；
- `Square`：file 0～8，rank 0～9；
- `Move`：From、To，可附带捕获、标记和规范字符串；
- `Position`：棋盘、轮到方、历史相关字段、哈希；
- `MoveRecord`：走前/走后 FEN、坐标着法、中文记谱、吃子、将军、耗时、哈希和序号；
- `GameResult` 与 `TerminationReason`；
- `RuleSet`：版本化休闲规则/未来竞赛规则。

公开构造函数必须校验坐标和状态，避免产生越界或缺失将帅的无效局面。

### 6.2 必须支持的规则

- 9 路 × 10 行棋盘；
- 帅/将九宫限制和将帅照面；
- 仕/士斜行与九宫限制；
- 相/象不过河和塞象眼；
- 马腿；
- 车直线阻挡；
- 炮的移动与炮架吃子；
- 兵/卒过河前后规则；
- 不允许走后己方仍被将军；
- 将军、应将、将死；
- 无合法着判负；
- 认输、超时、断线恢复、求和等业务终局状态。

### 6.3 局面变更

两种可选策略：

1. 不可变 `Position` 快照：正确性高，实现直接，但搜索中分配可能较多；
2. 受控 `MakeMove/UnmakeMove`：搜索性能更好，但必须保存完整 Undo 信息。

建议领域 API 对外表现为不可变，内置搜索内部使用严格封装的 `make/unmake`。Undo 数据需恢复：

- 棋子位置与被吃棋子；
- 轮到方；
- 哈希；
- 将军/历史信息；
- 重复局面计数；
- 其他规则集状态。

### 6.4 FEN、坐标与记谱

- 定义唯一规范 FEN 方言并在 `docs/rules.md` 记录；
- 服务端坐标固定，不随红黑视角翻转；
- 对外走法可同时返回结构化坐标和 ICCS 字符串；
- 中文纵线记谱仅用于显示和导入；
- 所有转换集中到领域模块；
- FEN 解析必须限制长度、字符集和棋盘结构；
- 中文记谱解析遇到歧义时返回候选和结构化错误，不静默猜测。

### 6.5 哈希与重复规则

- 使用固定随机表的 64 位或 128 位 Zobrist 哈希；
- 哈希至少包含棋子位置和轮到方，其他影响规则的状态也必须纳入；
- 随机表应稳定生成或固化，确保跨进程可复现；
- MVP 采用文档化休闲规则，例如三次重复局面可判和；
- 长将、长捉作为后续版本化规则集，在测试完整前不得宣称符合正式赛事全部规则。

## 7. 内置基础引擎

### 7.1 功能范围

- 合法走法生成；
- Negamax 或 Minimax + Alpha-Beta；
- 迭代加深；
- 静态评估：子力、位置表、机动性、将帅安全、兵卒结构；
- 吃子搜索/静态搜索；
- 置换表；
- 走法排序；
- `context.Context` 取消；
- 时间、深度和节点预算；
- MultiPV 基础支持或统一候选结果结构；
- 固定配置和固定随机种子下可复现。

### 7.2 结果结构

```go
type AnalyzeResult struct {
    BestMove Move
    Candidates []Candidate
    Depth int
    Nodes uint64
    Duration time.Duration
    PV []Move
    Score Score
    StoppedReason string
}
```

`Score` 应明确是红方视角还是当前行棋方视角，并能表达普通分值与 Mate 距离。该约定必须在 API 文档中固定。

### 7.3 搜索安全

- 每层迭代检查取消与预算；
- 超时返回最后一个完整深度结果；
- 置换表有内存上限；
- 搜索过程不修改共享权威局面；
- 搜索输入使用局面快照；
- 引擎候选写回前由规则层重新验证；
- benchmark 记录深度、节点数、NPS、内存和停止延迟。

## 8. 外部引擎适配

### 8.1 通用接口

```go
type Engine interface {
    Name() string
    Analyze(ctx context.Context, req AnalyzeRequest) (AnalyzeResult, error)
    BestMove(ctx context.Context, req MoveRequest) (MoveResult, error)
    Health(ctx context.Context) error
    Close() error
}
```

业务层依赖接口，不依赖 Pikafish 或具体进程结构。

### 8.2 UCI 适配器

优先实现独立 UCI 适配器，处理：

- 子进程启动、`uci` 握手、`isready`；
- `position`、`go`、`stop`、`quit`；
- `info`、PV、MultiPV、score、depth、nodes 解析；
- 受控配置：Threads、Hash、评估文件路径等；
- stdout 逐行读取，stderr 隔离；
- 请求级上下文取消和硬超时；
- 崩溃检测、重启、僵尸进程回收；
- 进程池并发上限、排队超时和健康检查；
- 关闭服务时优雅回收；
- 日志脱敏和输出长度限制。

UCCI 若后续需要，建立独立适配器，不与 UCI 状态机混写。

### 8.3 配置与安全

- 可执行文件路径仅从服务端配置白名单读取；
- 用户不能上传、替换或指定二进制；
- 引擎运行账户使用最小权限；
- 限制 CPU、内存、线程数和并发搜索数；
- 禁止把不可信引擎输出直接作为 SQL、日志字段或前端 HTML；
- 引擎不可用时自动降级到内置引擎或返回可恢复错误；
- 记录实际使用的引擎、版本、配置和降级原因。

## 9. 难度系统

### 9.1 DifficultyProfile

```text
engineType
moveTimeMs
maxDepth
maxNodes
multiPV
candidateEvalBand
samplingTemperature
maxAllowedEvalLoss
bookProbability
bookMinSamples
styleWeight
intentionalMistakeRate
```

数据库保存版本化配置，服务启动时加载并校验合理范围。对局创建后保存所使用配置快照，避免管理员调整配置后改变进行中的对局。

### 9.2 选着原则

- 所有候选必须合法；
- 高难度增加搜索资源、缩小候选评分带、降低温度与故意失误率；
- 低难度只在有限评分损失范围内采样，不随机送将或明显送大子；
- 棋风重排不能突破 `maxAllowedEvalLoss`；
- 固定种子可复现，正式对局使用安全随机种子；
- 未经过大量基准对局校准，不标注精确 Elo；
- 至少五档在资源与选着行为上具有稳定、可解释差异。

### 9.3 验证工具

提供 `cmd/tools/enginebench`：

- 固定局面集对各难度运行；
- 记录用时、深度、节点、候选差、失误分布和停止延迟；
- 支持固定随机种子；
- 可进行版本间回归比较；
- 后续可扩展难度之间的自动对局校准。

## 10. 对局服务与状态机

### 10.1 状态

建议至少包含：

```text
created
active_player_turn
active_ai_thinking
undo_pending
finished
aborted
recoverable_error
```

结束原因包括：将死、无合法着、认输、超时、同意和棋、重复规则、管理员终止或不可恢复错误。

### 10.2 创建对局

校验并固化：

- 用户执色；
- 初始局面；
- 难度配置快照；
- AI 模式；
- 学习版本；
- 棋风画像；
- 时间控制；
- 是否允许悔棋；
- 实际引擎选择和可用性。

若用户执黑，创建后应立即调度红方 AI 首着；HTTP 不长时间等待搜索完成。

### 10.3 玩家落子

事务流程：

1. 校验用户、对局状态、轮到方和客户端版本；
2. 解析规范坐标；
3. 由规则层验证合法性；
4. 原子写入 `match_moves` 和新对局快照/version；
5. 判断终局；
6. 若未结束，标记 AI 思考并提交异步任务；
7. 提交事务后发布 WebSocket 事件；
8. HTTP 快速返回“已接受/AI 思考中”。

重复请求通过幂等键和唯一约束返回同一结果，不重复落子。

### 10.4 AI 落子

严格执行：

1. 读取权威状态并校验轮次；
2. 查询启用学习库；
3. 达到样本和置信度要求时，按概率决定是否使用学习着；
4. 否则请求引擎 MultiPV；
5. 用规则层过滤候选；
6. 在允许评分损失内执行棋风重排和难度采样；
7. 再次验证最终走法；
8. 使用 expected version 原子提交；
9. 若版本过期则丢弃，不重放到新局面；
10. 推送走法、思考时间、引擎、是否命中学习库等事件。

### 10.5 悔棋、取消与恢复

- 悔棋请求先增加/切换对局版本并取消旧搜索；
- 根据产品规则回退一手或一回合，必须明确；
- 回退后重新计算局面哈希、重复计数和当前轮次；
- 旧 AI 结果即使稍后返回，也因版本不匹配被拒绝；
- WebSocket 断线不暂停服务端权威计时；
- 客户端重连通过快照和事件游标恢复；
- 服务重启后可从数据库 moves 重建对局，Redis 丢失不能损坏长期状态。

## 11. 棋谱导入与规范化

### 11.1 MVP 格式

- 中国象棋 PGN/文本棋谱；
- UTF-8 和常见中文编码的安全识别；
- 标准坐标着法；
- 常见中文纵线记谱；
- 项目自有 JSON 导入/导出。

XQF 等二进制格式延后，直到具备格式说明、多版本样本、损坏文件防护和完整测试。

### 11.2 上传安全

- 流式读取，不把大文件整体载入内存；
- 限制单文件大小、批次文件数、总大小、最大行长和最大着数；
- 校验扩展名、MIME、文件头和编码；
- 文件名仅作为显示信息，存储路径使用服务端生成 ID；
- 防止路径穿越、压缩炸弹和异常编码；
- 原始文件内容哈希和元数据入库；
- 对象存储接口负责隔离用户和生命周期；
- 不执行文件，不允许宏、脚本或可执行附件。

### 11.3 解析管线

```text
接收上传
 -> 安全校验
 -> 保存原始文件和摘要
 -> 拆分单盘棋局
 -> 解析元信息
 -> 从初始 FEN 逐着回放
 -> 每着规则校验
 -> 生成规范坐标/FEN/中文展示记谱
 -> 对规范着法序列计算内容哈希
 -> 去重并持久化
 -> 汇总批次报告
```

单盘失败不得导致整个批次回滚。错误结构至少包含：文件、局号、回合/ply、原文片段、错误码、原因和可选候选解释。

### 11.4 去重策略

- 原始文件哈希用于识别完全相同文件；
- 规范化着法序列哈希用于识别同一棋局的不同文本表现；
- 唯一范围可按全局或用户配置，但需明确隐私边界；
- 重复项保留来源关联和批次报告，不重复创建核心棋谱记录；
- 允许同一棋谱属于多个集合。

## 12. 学习任务管线

### 12.1 版本化原则

- 每次学习创建新的不可变 `learning_version`；
- 记录输入集合、参数快照、代码/算法版本、统计、错误和构建时间；
- 构建中的中间表与已发布版本隔离；
- 质量检查通过后才能激活；
- 激活操作原子切换当前版本；
- 回滚是重新激活旧版本，不修改旧版本内容；
- 删除原始棋谱时，需按数据保留策略决定已发布聚合统计是否保留。

### 12.2 局面着法库

针对每个规范局面和候选着保存：

- 学习版本；
- 局面哈希；
- 轮到方；
- 规范走法；
- 出现次数；
- 红胜、黑胜、和棋次数；
- 平滑评分/胜率；
- 近期权重；
- 来源集合/标签等可聚合信息；
- 样本量和置信度。

查询关键索引：`(learning_version_id, position_hash, side_to_move)`，唯一约束为 `(learning_version_id, position_hash, move)`。

哈希命中后应按需使用 FEN 或紧凑局面签名进行二次确认，规避极低概率哈希碰撞和实现错误。

### 12.3 棋风画像

提取可解释特征：

- 主动将军频率；
- 吃子、兑子倾向；
- 子力活动度；
- 进攻侧偏好；
- 开局体系偏好；
- 风险和复杂度偏好；
- 早出车、挺兵、兑车等特征；
- 平均对局长度和残局偏好。

画像需保存样本量、置信区间或置信等级。低置信度特征不能被包装为确定结论。

### 12.4 质量报告

至少包含：

- 有效棋谱数和有效着数；
- 重复率、非法着率；
- 覆盖局面数；
- 主要开局分布；
- 红胜、黑胜、和棋分布；
- 低样本局面数量；
- 棋风统计置信度；
- 构建耗时、失败和警告；
- 使用的算法与参数版本。

### 12.5 任务恢复

- 大批量数据分块处理并保存 checkpoint；
- 10,000 盘棋谱不要求单事务完成；
- worker 崩溃后从安全检查点恢复；
- 每个步骤幂等；
- 失败重试使用退避并设置最大次数；
- 永久失败进入可诊断状态，不无限循环；
- 版本发布前执行数量、索引和抽样回放校验。

## 13. 复盘与用户自适应

### 13.1 异步复盘

对局结束后创建 `analysis_job`：

1. 按关键节点或全局配置逐着调用引擎；
2. 保存实战着评分、最佳着评分和评分损失；
3. 保存有限数量 MultiPV 和主变化；
4. 标记关键转折、漏着、失误和严重失误；
5. 关联开局库命中和 AI 学习着使用情况；
6. 生成评分曲线和文本摘要所需结构化数据。

原始引擎日志和大量 PV 不得无界写入主表，应限制长度、压缩或存入冷存储。

### 13.2 用户统计

MVP 只做数据采集和可解释统计：

- 重复失误局面；
- 薄弱开局；
- 常见漏算类型；
- 不同难度、执色和模式下胜率；
- 平均思考时间和对局长度。

不做黑盒在线训练。后续专项练习应基于可追溯规则和复盘数据生成。

### 14.1 主要实体

- `users`；
- `record_files`；
- `record_import_batches`；
- `game_records`；
- `record_moves`；
- `positions`；
- `record_collections`；
- `collection_records`；

### 14.2 关键约束

- 规范化棋谱内容哈希唯一或按用户唯一；
- `match_moves(match_id, ply)` 唯一；
- `matches.version` 单调递增；
- `opening_book_entries(learning_version_id, position_hash, move)` 唯一；
- 学习版本发布后核心统计不可更新；
- 任务状态转换受约束，不能从终态任意回到运行态；
- 所有用户数据查询包含租户/用户隔离条件；

- 时间统一存 UTC；
- 大文本、PV、日志设置长度和保留期。

### 14.3 事务边界

- 玩家/AI 落子：对局校验、move 插入、version 更新和状态变更同一事务；
- 启用学习版本：质量状态校验和激活指针切换同一事务；
- 创建任务：业务记录与 Outbox 同一事务；
- 上传批次：批次可先创建，单盘记录分事务处理；
- 复盘结果：按对局/分块提交，任务进度单独更新，支持恢复。

### 15.2 请求与错误规范

- JSON 字段使用稳定命名；
- 写接口支持 `Idempotency-Key`；
- 客户端提交 `expectedMatchVersion`；
- 错误响应包含 `code`、`message`、`requestId`、可选 `details`；
- 409 表示版本/状态冲突；
- 422 表示语义或非法走子；
- 429 表示限流；
- 503 表示依赖不可用且无法降级；
- 不向客户端返回堆栈、SQL、文件系统路径或原始引擎命令行。

### 15.3 WebSocket 事件

统一事件信封：

```json
{
  "eventId": "...",
  "matchId": "...",
  "matchVersion": 12,
  "type": "match.move_applied",
  "timestamp": "2026-07-15T01:00:00Z",
  "payload": {}
}
```

事件类型建议：

- `match.snapshot`；
- `match.move_accepted`；
- `match.move_rejected`；
- `match.ai_thinking`；
- `match.ai_move_applied`；
- `match.undo_applied`；
- `match.finished`；
- `match.engine_degraded`；
- `job.progress`；
- `job.completed`；
- `job.failed`。

事件可重复投递，客户端按 `eventId` 和版本去重。重连必须支持获取最新快照和缺失事件，不能假设 WebSocket 永不丢包。

## 16. 幂等、并发与任务队列

### 16.1 幂等

- 创建对局、落子、悔棋、认输、创建学习任务、启用版本和创建复盘均定义幂等策略；
- 幂等记录关联用户、路由、键、请求摘要、响应摘要和过期时间；
- 同一键不同请求体应拒绝；
- 数据库唯一约束作为最终防线。

### 16.2 并发

- 对局使用乐观锁：`UPDATE ... WHERE id=? AND version=?`；
- AI 任务携带 expected version；
- 同一对局同一版本只允许一个有效 AI 搜索；
- 悔棋和认输可取消搜索；
- 多实例部署时使用数据库/Redis 协调，但不能只依赖进程内锁；
- 引擎池按用户和全局配额调度，防止单用户耗尽 CPU。

### 16.3 队列

队列任务类型：

- AI 落子；
- 棋谱解析/导入；
- 学习版本构建；
- 对局复盘；
- 数据导出/清理。

任务信封包含任务 ID、类型、版本、用户、尝试次数、幂等键、创建时间和 trace 信息。Redis 不可用时：

- 开发环境允许内存队列；
- 生产环境可拒绝新的重任务并保持现有权威数据安全；
- 基础低难度 AI 可按受控策略使用进程内执行，但必须有超时和并发限制；
- 恢复后扫描数据库中的待处理任务补投递。

## 17. 安全方案

- 上传使用流式限制、格式校验和解析超时；
- 防止压缩炸弹、路径穿越、异常编码、超长行和超大着数；
- 外部引擎路径只来自白名单配置；
- 每次搜索有上下文取消、硬超时、节点/线程/内存配额；
- 上传、创建对局、分析和学习任务限流；
- 所有用户资源执行所有权校验；
- 写操作记录用户、请求 ID、时间、对象、前后状态摘要和结果；
- 配置通过环境变量/密钥管理注入，不提交真实密钥；
- SQL 使用参数化查询；
- 日志脱敏，不记录凭证、完整用户文件或不受限引擎输出；
- 生产不暴露 `pprof`、内部 Prometheus 或调试端点；
- 数据导出按用户隔离；
- 删除、保留和备份策略形成文档。

若引入用户登录，建议使用成熟身份方案或安全的服务端会话，不自行设计密码学协议。

## 18. 可观测性与运维

### 18.1 日志

结构化字段建议：

- request/trace ID；
- user ID（脱敏）；
- match/job/import ID；
- match version；
- engine name/profile；
- duration、result、error code；
- 降级和重试原因。

规则层单测不依赖日志；生产日志不输出完整 PV 或用户棋谱正文。

### 18.2 指标

- HTTP 请求量、延迟、状态码；
- WebSocket 连接、重连和推送失败；
- 活跃对局和落子冲突；
- AI 队列长度、等待时间、搜索耗时、超时、取消；
- 内置/外部引擎成功率和降级次数；
- 棋谱导入成功、重复、失败、解析耗时；
- 学习任务进度、失败、构建耗时；
- 复盘队列和处理耗时；
- MySQL/Redis 连接池和错误；
- 进程池数量、重启和僵尸回收。

### 18.3 健康检查

- liveness：进程主循环是否存活；
- readiness：MySQL、迁移状态、必要配置是否就绪；
- Redis、对象存储和外部引擎分别报告可用/降级，不应一概使 API 不存活；
- 引擎健康接口返回安全摘要，不泄露路径或命令行。

## 19. 测试方案

### 19.1 规则单元测试

覆盖所有棋子正常、边界、阻挡、吃子和非法走法，重点包括：

- 马腿、象眼、炮架；
- 将帅照面；
- 自陷将军；
- 将军、应将、将死、无合法着；
- 红黑坐标转换；
- FEN 解析/序列化往返；
- 中文记谱歧义；
- `make/unmake` 后棋盘、哈希、轮到方和历史完全恢复；
- 三次重复休闲规则；
- 非法或畸形局面的拒绝。

### 19.2 属性、模糊与基准测试

- 所有生成走法都能通过合法性校验；
- 任意合法局面走一步再撤销后状态不变；
- FEN round-trip；
- 解析器面对随机/损坏输入不崩溃、不超限；
- 固定种子下难度采样可复现；
- 规则、哈希、走法生成、搜索建立 benchmark；
- perft 数据只使用可信夹具并注明来源，不编造计数。

### 19.3 服务集成测试

- 创建对局、并发/重复落子、版本冲突；
- AI 结果过期、取消和悔棋；
- 外部引擎握手、MultiPV、超时、崩溃和恢复；
- Redis 不可用的降级；
- MySQL 事务回滚和唯一约束；
- 损坏、乱码、重复和部分非法棋谱导入；
- 学习任务 checkpoint、重试、发布和回滚；
- 复盘失败恢复；
- 用户数据隔离、上传限制和限流。

### 19.4 E2E

与前端共同覆盖：

1. 创建对局 → 玩家走子 → AI 回应 → 悔棋；
2. 上传棋谱 → 导入报告 → 创建学习版本 → 启用；
3. 以学习库开始对局 → 命中学习着或可靠回退引擎；
4. 完成对局 → 异步复盘 → 查看关键转折；
5. 外部引擎崩溃 → 对局不损坏 → 降级或恢复；
6. 服务/连接短暂中断 → 客户端恢复权威状态。

## 20. 性能目标与验证

MVP 目标：

- 普通 API 本地开发环境 P95 < 200ms，不含引擎搜索和大文件分析；
- 玩家落子完成规则验证和事务提交后快速返回，不同步等待 AI；
- 搜索严格服从时间/节点预算，取消延迟可测；
- 10,000 盘棋谱支持分批、异步、checkpoint 和恢复；
- 学习库按版本和局面查询有复合索引；
- 大 PV、日志和任务历史有大小/保留限制；
- 使用 benchmark、压测和指标证明优化。

建议压测场景：

- 普通读 API；
- 同一对局重复/并发落子；
- 多对局同时触发 AI；
- 引擎池满载和排队超时；
- 大批棋谱导入；
- 热门学习版本的局面查询；
- WebSocket 多连接广播与重连。

## 22. 分阶段实施计划

### 阶段 B0：设计与脚手架

交付：

- monorepo 和 Go 模块；
- `cmd/api`、`cmd/worker` 最小入口；
- 配置、日志、健康检查；
- MySQL/Redis Docker Compose；
- 初始迁移框架；
- API 草案、架构文档和 ADR；
- CI 与开发命令。

验收：一条命令启动环境，健康检查可用，测试可运行。

### 阶段 B1：规则核心

交付：

- 棋盘、棋子、坐标、Move；
- FEN 解析/序列化；
- 所有棋子基础走法与吃子；

### 阶段 B4：棋谱导入与学习

交付：

- 流式上传和对象存储；
- 文本/PGN/坐标/中文记谱/JSON 解析；

- 导入批次和逐项错误报告；
- 棋谱集合；
- 异步学习任务、checkpoint；
- 局面着法库、棋风画像、质量报告；
- 不可变学习版本、启用和回滚；
- 对局中的学习命中、棋风重排和引擎回退。

验收：完成“上传 → 报告 → 构建版本 → 启用 → 对局命中/回退”闭环。

### 阶段 B5：复盘与产品化

交付：

- 异步复盘和 MultiPV 分析；
- 评分曲线数据、关键转折和失误统计；
- 用户薄弱开局/重复失误统计；
- 历史、导出、清理和保留策略；
- 安全、限流、可观测性、性能压测和部署收尾；
- 完整 README、运维和许可证文档。

验收：满足需求文档中的后端相关 Definition of Done。

## 23. 首轮建议执行范围

首次编码不要同时铺开全部模块，建议停在以下可运行节点：

1. 初始化 Go monorepo、API、worker、配置、日志、健康检查和 CI；
2. 编写架构 ADR、坐标约定和规则范围文档；
3. 实现 `internal/domain/xiangqi` 的核心类型；
4. 实现初始 FEN 解析和序列化；
5. 实现基础棋子走法、边界/阻挡和第一批合法性校验；
6. 添加马腿、象眼、炮架、将帅照面等关键单测；
7. 提供 `go test ./...`、运行 API 和 Docker Compose 命令；
8. 在测试通过后报告已完成、未完成、限制与下一阶段，不提前伪造引擎和学习功能。

## 24. 后端完成标准

后端 MVP 完成需同时满足：

- 服务端拒绝所有非法着并正确处理关键规则边界；
- 可创建、进行、恢复和结束人机对局；
- 至少五档难度有稳定、可解释的资源和选着差异；
- 内置引擎可停止、可测、可复现；
- 外部引擎可选，异常时可降级或返回可恢复错误；
- 棋谱可安全导入并得到逐项成功/重复/失败报告；
- 可生成、启用和回滚版本化学习库；
- AI 命中学习局面时按配置使用，未命中时可靠回退；
- 可生成基础复盘和学习库命中统计；
- 10,000 盘棋谱任务支持异步、分批和恢复；
- 普通 API、搜索预算、并发和任务性能达到目标并有指标证明；
- Docker Compose 可启动开发环境；
- Go 测试、静态检查、集成测试、前端联调和构建通过；
- 无硬编码密钥、假训练结果、越权数据访问或未说明的许可证风险。

## 25. 主要风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 象棋规则边界错误 | 非法走子、错误胜负 | 纯领域层、可信夹具、属性测试、perft 和版本化规则集 |
| 坐标/FEN/记谱多套实现 | 导入、UI、引擎结果不一致 | 所有转换集中在领域模块，共享契约 |
| make/unmake 状态未完全恢复 | 搜索结果随机错误 | Undo 完整建模、状态哈希断言和属性测试 |
| 外部引擎挂死或输出异常 | 请求阻塞、资源泄漏 | 硬超时、取消、输出限制、进程池和僵尸回收 |
| AI 结果写入过期局面 | 悔棋后错误落子 | expected version + 事务乐观锁 + 结果二次验证 |
| 大批棋谱拖垮服务 | 内存/CPU/数据库压力 | 流式上传、分块任务、checkpoint、配额和限流 |
| 中文记谱歧义 | 错误规范化和学习污染 | 返回结构化歧义，不静默猜测，逐着规则验证 |
| 哈希碰撞或实现错误 | 学习库误命中 | 足够位宽、固定测试、命中后二次局面校验 |
| 学习版本数据不完整 | 对局行为不可追溯 | 不可变版本、质量门禁、原子激活和回滚 |
| Redis 成为唯一状态 | 缓存丢失导致对局损坏 | MySQL 权威、Redis 可重建、恢复扫描 |
| 难度“随机犯错”违和 | 用户体验差、难度失真 | MultiPV 候选带、最大评分损失和基准校准 |
| Pikafish 许可证处理不当 | 分发法律风险 | 独立进程、记录版本/来源、确认 GPLv3 分发策略 |

---

后端实施必须坚持总体优先级：**规则正确性 → 对局状态机 → 可控引擎 → 棋谱规范化 → 局面库与棋风重排 → 复盘**。任何高级 AI 能力都不能绕过规则、数据质量、可测试性和许可证约束。
