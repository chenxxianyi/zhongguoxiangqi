> 文档日期：2026-07-16  
> 适用范围：Vue 3 前端、Go 后端、象棋规则引擎、实时对局、棋谱学习与复盘模块

- 已修复逐着中文记谱，实际着法和最佳着法均使用对应 `MoveRecord.fenBefore`。
- 已细分 `ILLEGAL_MOVE`、`MATCH_VERSION_CONFLICT`、`MATCH_NOT_FOUND` 和网络错误。

- 已修正进行中与已结束对局的跳转目标。
- 已将首页棋谱数量、学习版本和激活状态接入真实接口。

- 生产环境禁止 MySQL 失败后静默降级。
- 多实例共享事件总线及游标超出保留历史时的显式恢复事件。
- 统一偏好 Store、真实音效、棋盘坐标和减少动态效果。
- 快速/深度分析档位、可交互复盘棋盘和 Playwright 端到端测试。

下一建议批次保持为“可靠对局”：优先实现 MySQL 权威对局仓库、事务幂等和生产启动
失败策略；随后再进入设置与对弈体验优化。

## 1. 文档目的

当前项目已经完成中国象棋应用的核心 MVP，具备以下主要能力：

- 创建人机对局并选择执色、难度和 AI 模式
- 后端权威规则校验和异步 AI 行棋
- REST 与 WebSocket 前后端联调
- 棋子移动、吃子、悔棋及落点动画
- 对局历史、棋谱导入、学习版本和复盘分析
- 前后端单元测试、类型检查、Lint、构建和 CI
- MySQL 连接、数据库初始化和迁移基础

下一阶段不宜继续优先增加装饰性页面或复杂视觉效果，而应重点解决以下问题：

1. 前端提示与后端规则的一致性。
2. 对局、结果和复盘信息的准确性。
3. 服务重启和网络波动下的数据可靠性。
4. 设置、音效等功能的真实可用性。
5. 自动化测试对完整用户流程的覆盖。

最终目标是将项目从“功能完整的 MVP”提升为“可以稳定长期使用和部署的象棋应用”。

## 2. 优化原则

### 2.1 后端保持权威

棋局合法性、当前行棋方、胜负结果、重复局面和终局判定必须以后端规则引擎为准。前端可以进行预览和交互优化，但不能维护一套可能与后端不一致的规则。

### 2.2 服务端状态与动画状态分离

服务端 FEN、版本号和着法历史是权威状态；棋盘动画只负责表现服务端确认后的变化，不反向影响业务状态。

### 2.3 先保证正确，再提升体验

优化优先级按以下顺序执行：

1. 数据和规则正确。
2. 网络和持久化可靠。
3. 用户操作可恢复。
4. 交互反馈清楚。
5. 视觉和性能精修。

### 2.4 每项优化都应可验证

每个阶段必须同时包含：

- 明确的接口或行为定义
- 单元测试
- 集成或端到端测试
- 错误状态和恢复路径
- 可观察的验收标准

## 3. 当前主要问题

## 3.1 前端候选着法是简化算法（已完成）

优化前，前端在 `apps/web/src/utils/board.ts` 中自行生成候选落点，但没有完整处理：

- 马腿阻挡
- 象眼阻挡
- 象不能过河
- 士、将、帅的九宫限制
- 车、炮的路径阻挡
- 炮吃子时的炮架数量
- 目标位置是否存在己方棋子
- 将帅照面
- 走后己方是否仍被将军

当前简化算法已经删除，生产落点改为读取后端权威合法着法接口。

## 3.2 胜负显示没有完全结合玩家执色（已完成）

首页、历史、对局和复盘页面已统一使用玩家视角结果工具，未结束对局不会被计为失败。

## 3.3 复盘记谱使用了错误的局面上下文（已完成）

逐着复盘和对局着法列表已改为使用每条着法自己的 `fenBefore`。实际着法与最佳着法
使用相同的起始局面转换。

## 3.4 MySQL 尚未成为权威仓库

项目已经具备 MySQL 连接和迁移能力，但对局服务仍固定使用内存仓库。当前可能存在以下问题：

- 服务重启后对局丢失
- 幂等请求记录丢失
- 分析任务和结果丢失
- 学习任务与激活版本丢失
- 多实例部署时状态不一致

## 3.5 WebSocket 重连可能漏事件（部分完成）

前端现已记录最后成功消费的事件 ID，并在重连时携带 `afterEventId`；发现版本跳跃时
会获取完整快照。仍待完成游标超出单机历史、多实例共享事件总线等生产能力。

## 3.6 设置页部分控件没有真实业务效果

以下设置目前没有形成统一、持久化的用户偏好：

- 默认棋盘方向
- 显示棋盘坐标
- 落子音效
- 减少动态效果

棋盘工具栏中的音效按钮目前主要改变状态和提示文字，没有播放真实音效。

## 3.7 复盘分析预算偏低

当前分析使用较低搜索深度和较短时间预算，适合快速预览，但不足以稳定支持“大漏”“失误”等强结论。

## 3.8 生产运行时静态或占位数据（已完成）

前端生产代码中的 Demo 棋局、固定统计、虚构用户、静态难度兜底和未实现的数据操作
已经移除。首页、历史、棋谱、学习、分析、设置和侧栏均读取后端接口；接口失败时显示
明确不可用状态，不再把失败伪装成 `0`、空列表或默认业务数据。独立 `h5-demo` 静态
原型也已从仓库删除，测试文件中的 fixture/mock 继续保留且不进入生产构建。

## 3.9 缺少完整端到端测试

现有单元测试和 CI 已经具备良好基础，但还没有覆盖真实浏览器中的完整对局流程、断线恢复和响应式交互。

## 5. 第一阶段：对局可信度优化

## 5.1 后端提供合法着法接口

### 建议接口

```http
GET /api/v1/matches/{id}/legal-moves
```

返回当前行棋方的全部合法着法：

```json
{
  "matchId": "match-id",
  "matchVersion": 12,
  "sideToMove": "red",
  "moves": [
    {
      "move": "a3a4",
      "from": "a3",
      "to": "a4",
      "capture": false
    }
  ]
}
```

也可以按起点查询：

```http
GET /api/v1/matches/{id}/legal-moves?from=a3
```

### 后端要求

- 必须复用 `internal/domain/xiangqi` 的权威规则。
- 返回结果必须绑定 `matchVersion`。
- 非玩家回合可以返回空数组或明确的状态错误。
- 对局已结束时不得返回可执行着法。
- 不应由 HTTP 层重新实现规则。

### 前端调整

- 选中棋子时读取服务端合法着法。
- 以 `matchVersion + fromSquare` 作为短期缓存键。
- 版本变化后清空缓存。
- 请求期间显示选中状态，但不要显示未经确认的落点。
- 网络失败时可以不显示提示，但仍允许用户提交，由后端最终判断。
- 删除或仅在离线演示中保留简化的 `candidateMoves`。

### 验收标准

- 前端显示的每个落点都能被后端接受。
- 马腿、象眼、炮架、将帅照面等规则有自动化测试。
- 吃己方棋子、自陷将军等位置不会显示为可走。

## 5.2 修复胜负结果显示

建议提取统一工具函数：

```ts
getPlayerResult(outcome, playerColor)
```

统一返回：

```ts
{
  type: 'win' | 'loss' | 'draw' | 'ongoing',
  label: '胜' | '负' | '和' | '进行中'
}
```

所有页面必须复用同一函数：

- 首页最近对局
- 历史对局
- 对局结束提示
- 复盘结果
- 统计胜率

### 验收标准

- 玩家执红获胜显示“胜”。
- 玩家执黑获胜显示“胜”。
- 玩家执红或执黑失败均显示“负”。
- 未结束对局不显示为失败。

## 5.3 修复逐着中文记谱

复盘页面应使用每条着法自己的：

- `move`
- `side`
- `fenBefore`

而不是当前对局最终 FEN。

建议在分析结果中保留或关联原始 `MoveRecord`，由统一的记谱工具完成转换。

### 验收标准

- 同一路线上有两个车、炮或兵时，中文记谱仍正确。
- 红黑双方的纵线编号方向正确。
- 实际着法和最佳着法使用同一个起始局面转换。

## 5.4 细化着法提交错误

前端不应把所有失败统一显示为“着法提交失败”。

建议根据 `ApiError.body.code` 区分：

| 错误码 | 用户提示 | 前端动作 |
|---|---|---|
| `ILLEGAL_MOVE` | 该着法不合法 | 恢复选择并播放非法着法反馈 |
| `MATCH_VERSION_CONFLICT` | 局面已更新，正在同步 | 重新获取最新快照 |
| `MATCH_NOT_FOUND` | 对局不存在或已失效 | 返回历史或新建对局 |
| 网络错误 | 网络连接异常，请重试 | 保留当前棋盘并提供重试 |

## 6. 第二阶段：持久化与实时可靠性

## 6.1 抽象对局仓库接口

当前 `game.Service` 直接依赖 `MemoryRepository`。建议改为依赖接口：

```go
type Repository interface {
    Create(match Match) error
    Get(id string) (Match, error)
    List() []Match
    Update(id string, expectedVersion int64, update func(*Match) error) (Match, error)
    Idempotency(route, key, digest string) (Match, bool, error)
    SaveIdempotency(route, key, digest string, match Match)
}
```

随后实现：

- `MemoryRepository`
- `MySQLRepository`

启动时根据 `XIANGQI_DATA_MODE` 注入对应实现。

## 6.2 MySQL 数据持久化范围

至少应持久化：

- 对局基础信息
- 当前 FEN
- 对局版本
- 着法历史
- 对局结果和终止原因
- 幂等请求记录
- 棋谱及导入批次
- 学习任务、版本和激活状态
- 分析任务与分析结果

### 事务要求

一次着法更新应在同一事务内完成：

1. 校验当前版本。
2. 写入着法。
3. 更新对局 FEN、状态和版本。
4. 保存幂等结果。
5. 提交事务。

避免出现“对局已更新但着法记录未写入”的中间状态。

## 6.3 禁止生产环境静默降级

当前 MySQL 连接失败后会退回内存模式。建议：

- 开发环境可以允许显式降级。
- 生产环境配置为 MySQL 时，连接失败应直接停止启动。
- `/health/ready` 应真实检查数据库和引擎状态。
- 健康接口返回实际 `authoritativeStore`，不能固定显示 `memory`。

## 6.4 WebSocket 事件游标

前端应记录最后成功处理的 `eventId`：

```ts
let lastEventId: string | null = null
```

重连 URL：

```text
/matches/{id}/stream?afterEventId={lastEventId}
```

### 恢复策略

1. 正常重连时请求游标后的事件。
2. 收到事件后先校验 `matchVersion`。
3. 发现版本跳跃时停止直接应用事件。
4. 调用 `GET /matches/{id}` 获取完整快照。
5. 以快照重新同步动画和业务状态。

### 后端增强

- 当 `afterEventId` 不在保留历史中时，明确通知客户端需要快照。
- 多实例部署时，事件总线不能只存在于单机内存。
- 后续可使用 Redis Streams、消息队列或数据库事件表。

### 验收标准

- AI 思考期间断网，恢复后棋盘能显示 AI 已完成的走法。
- 重复事件不会产生重复着法。
- 漏失事件时能通过快照恢复。
- 页面刷新后可以继续未结束对局。

## 7. 第三阶段：设置与对弈体验

## 7.1 建立统一偏好 Store

建议新增 `preferences` Store，管理：

- `boardOrientation`
- `showCoordinates`
- `soundEnabled`
- `reducedMotion`
- `theme`
- 可选的音量

设置应保存到 `localStorage`，并在应用启动时恢复。

### 注意事项

- `reducedMotion` 默认跟随系统 `prefers-reduced-motion`。
- 用户手动选择后可覆盖系统值。
- 设置页和棋盘工具栏操作同一份状态。
- 不要在 `match` Store 中保存跨对局偏好。

## 7.2 棋盘坐标

增加标准 ICCS 坐标显示：

- 文件：`a` 到 `i`
- 横线：`0` 到 `9`
- 翻转棋盘时坐标同步翻转
- 移动端不得遮挡棋子或棋盘线

## 7.3 真实音效

建议提供三个短音效：

- 普通落子
- 吃子
- 将军或终局

实现要求：

- 首次用户交互后再初始化音频。
- 使用短促、克制的本地资源。
- 音效不能延迟棋盘动画。
- 快速连续事件应限制叠加数量。
- 尊重静音设置。

## 7.4 将军和终局反馈

后端快照或事件中建议增加：

```json
{
  "inCheck": true,
  "checkedSide": "black"
}
```

前端表现：

- 被将军一方的将或帅显示一次明确但克制的警示。
- 不依赖 Toast 作为唯一反馈。
- 终局后禁用所有落子交互。
- 明确展示胜负和终止原因，例如将死、困毙、认输、和棋。

## 7.5 历史对局跳转

- 进行中的对局进入 `/match/{id}`。
- 已结束的对局进入 `/analysis/{id}`。
- 分析尚未完成时显示任务状态。
- 不应让未结束的对局直接进入复盘。

## 7.6 首页使用真实学习数据

首页学习区应读取：

- 已导入棋谱数量
- 学习版本数量
- 当前激活版本
- 最近构建状态

删除固定的 `0 盘` 和 `0`。

## 8. 第四阶段：复盘分析升级

## 8.1 分析档位

建议增加两种模式：

### 快速分析

- 用于对局结束后的即时反馈
- 每步较短时间预算
- 展示趋势和明显错误

### 深度分析

- 用户主动触发
- 更高深度、节点数和时间预算
- 支持后台任务
- 结果持久化，可重复查看

## 8.2 结果可信度

分析结果建议增加：

- 实际搜索深度
- 节点数
- 分析耗时
- 候选着数量
- 评分来源
- 分析档位

对于分析不足或实际着法不在候选集合中的情况，避免直接判断为“大漏”，可以显示：

- 未进入候选
- 分析深度不足
- 建议进行深度分析

## 8.3 复盘棋盘

逐着分析页面建议增加可交互棋盘：

- 上一步、下一步
- 跳转到指定回合
- 实际着法箭头
- 最佳着法箭头
- 当前局面评分
- 红黑双方视角切换

复盘棋盘只读取历史 FEN，不应连接实时对局事件流。

## 9. 第五阶段：测试体系

## 9.1 前端单元测试

重点覆盖：

- 玩家执黑时的胜负显示
- 每一步使用 `fenBefore` 转换记谱
- 合法着法接口缓存失效
- WebSocket 事件去重
- 版本跳跃后的快照恢复
- 设置持久化
- 音效启停
- 棋盘动画队列
- 悔棋恢复被吃棋子

## 9.2 后端单元与集成测试

重点覆盖：

- 全部棋子合法着法生成
- 将帅照面和自陷将军
- MySQL 乐观锁版本冲突
- 幂等键重复请求
- 事务回滚
- 数据库重启后的对局恢复
- 事件游标恢复
- 分析任务持久化

## 9.3 浏览器端到端测试

建议引入 Playwright，至少包含以下流程：

1. 创建红方对局并完成一次普通走子。
2. 创建黑方对局并等待 AI 先行。
3. 完成一次吃子。
4. 提交非法着法并查看反馈。
5. AI 快速返回时不丢动画或状态。
6. WebSocket 断开后自动恢复。
7. 悔棋撤回一到两步。
8. 刷新页面继续未结束对局。
9. 对局结束后进入复盘。
10. 桌面和移动端棋盘没有溢出、错位和遮挡。

## 10. 性能与可观察性

## 10.1 前端性能

- 棋子移动继续使用 `transform`，避免频繁布局。
- 只在正在移动的棋子上使用 `will-change`。
- 着法列表过长时使用虚拟列表或分页。
- 避免重复解析同一个 FEN。
- 合法着法和复盘局面可按版本缓存。
- 页面离开后清理轮询、计时器、WebSocket 和音频资源。

## 10.2 后端性能

- 为对局 ID、版本、更新时间建立索引。
- 着法历史采用按对局和步数排序的联合索引。
- 分析任务设置并发上限和超时。
- 对上传棋谱设置数量、大小和解析时间限制。
- 增加数据库连接池指标。
- 使用性能基准验证引擎难度配置。

## 10.3 可观察性

建议增加：

- 请求总量、错误率和延迟
- 当前 WebSocket 连接数
- AI 搜索耗时、深度和节点数
- 分析任务队列长度
- 数据库连接池状态
- 棋谱导入成功率
- WebSocket 重连和快照恢复次数

日志应继续携带：

- `requestId`
- `matchId`
- `matchVersion`
- `eventId`
- `jobId`

禁止在日志中输出数据库密码和完整敏感配置。

## 11. 安全与部署

## 11.1 HTTP 安全

- 生产环境限制允许来源。
- 增加请求体大小限制和超时。
- 对创建对局、分析任务、棋谱导入增加速率限制。
- 上传文件按实际内容校验，不能只依赖扩展名。
- 返回统一错误信封，不暴露内部堆栈。

## 11.2 配置安全

- 生产环境禁止使用默认数据库密码。
- 启动时校验关键配置。
- `.env` 不进入版本库。
- 开发、测试和生产使用独立配置。

## 11.3 部署要求

- API 无状态化之前不建议直接横向扩容。
- 使用 MySQL 后仍需解决事件总线和异步任务的多实例一致性。
- 优雅关闭时应停止接收新任务，并等待或取消 AI 搜索。
- 数据库迁移应有明确版本，不应只根据某张表是否存在判断全部迁移完成。

## 12. 建议实施计划

## 里程碑 A：准确性修复

预计范围：

- 修复逐着中文记谱
- 细化接口错误提示

- 复盘记谱基于每步 `fenBefore`。
- 相关单元测试通过。

## 里程碑 B：权威合法着法

预计范围：

- 后端合法着法查询能力
- 前端合法落点接入
- 版本缓存和错误恢复
- 完整规则测试

完成标准：

- 不再依赖简化规则生成生产落点。

## 里程碑 C：可靠对局

预计范围：

- 仓库接口抽象
- MySQL 对局仓库
- 事务和幂等持久化
- WebSocket 游标重连
- 快照恢复

- 网络断开后不漏 AI 着法。
- 版本冲突可以自动恢复。

## 里程碑 D：体验补全

预计范围：

- 设置 Store
- 棋盘坐标
- 真实音效
- 减少动态效果
- 将军和终局反馈

完成标准：

- 音效和动画设置在所有页面一致生效。

## 里程碑 E：复盘与质量保障

预计范围：

- 快速/深度分析
- 可交互复盘棋盘
- Playwright 端到端测试
- MySQL 集成测试
- 监控指标

完成标准：

- 核心对局流程进入 CI。
- 复盘结果包含明确的分析预算和可信度信息。

## 13. 推荐的首个开发批次

建议下一个开发批次控制在以下范围：

1. 修复玩家执黑时的胜负显示。
2. 修复复盘逐着记谱的 FEN 上下文。
3. 细分非法着法、版本冲突和网络错误。
4. 调整历史对局跳转规则。
5. 首页学习数据接入真实接口。
6. 为以上行为补充单元测试。

这一批改动范围较小、风险可控，并能立即提升用户对项目数据准确性的信任。

第二个批次再集中实现“后端权威合法着法”，避免同时改动过多规则、接口和持久化代码。

## 14. 总体验收标准

项目完成本轮优化后，应满足：

- 前端显示的可走位置全部由权威规则确认。
- 红黑双方胜负、回合和复盘记谱准确。
- 服务重启后对局和分析结果不丢失。
- 网络断开后能够自动恢复最新局面。
- 重复接口响应和重复事件不会重复走棋。
- 设置、音效、坐标和减少动态效果真实生效。
- 首页、历史、棋谱、学习和复盘均使用真实数据。
- 快速分析与深度分析有明确边界。
- 核心对局流程拥有浏览器端到端测试。
- 桌面和移动端均可稳定完成一整盘对局。

## 15. 结论

项目目前已经具有良好的产品结构、前后端基础和视觉完成度。后续最有价值的工作不是继续扩展页面数量，而是收紧规则、数据、实时通信和持久化之间的边界。

推荐严格按照以下顺序推进：

1. 准确性修复。
2. 权威合法着法。
3. MySQL 持久化和断线恢复。
4. 设置与对弈反馈。
5. 深度复盘和端到端测试。
6. 性能、监控和生产部署。

完成以上优化后，项目将具备从本地 MVP 向稳定部署版本演进的基础。
